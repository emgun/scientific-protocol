# Migrating to protocol 0.3.0

Version 0.3.0 is a deliberate pre-1.0 breaking release. It changes non-upgradeable contract
constructors, function signatures, returned structs, and generated bindings. It is not an in-place
upgrade of a 0.2.x deployment.

## Required operator actions

1. Preserve the existing deployment file, database backup, indexed block cursor, and service image
   digest as rollback evidence.
2. Deploy the complete 0.3.0 contract set. `BondEscrow` must be deployed after
   `ReplicationRegistry` and `ProtocolTreasury` because its constructor binds both immutably. Before the
   deployment administrator renounces authority, call
   `ClaimRegistry.configureProtocolDependencies(bondEscrow, replicationRegistry)` exactly once.
3. Publish a new deployment manifest with the new addresses and deployment block. Do not edit or
   relabel the 0.2.x manifest.
   Remote deployment requires exactly one explicit `SP_MIN_AUTHOR_BOND_ETH` or
   `SP_MIN_AUTHOR_BOND_WEI`. The deployer writes it to `ProtocolParameters` before surrendering
   parameter authority and records the same wei value as `parameters.minimumAuthorBondWei` in the
   manifest. A remote zero or omitted value fails deployment.
4. Stop the old write-enabled service, back up Postgres, and run:

   ```bash
   docker run --rm \
     --env-file /absolute/path/service.env \
     ghcr.io/emgun/scientific-protocol-service@sha256:<0.3.0-digest> migrate
   ```

5. Start the same immutable image digest in `SP_SERVICE_MODE=read-only`, sync from the new
   deployment block, and require `/readyz` to return HTTP 200.
6. Compare claim, artifact, replication, checkpoint, agent, and governance reads with direct chain
   reads. Then start a separately credentialed `write-enabled` instance if writes are required.
7. Update TypeScript consumers to `scientific-protocol@0.3.0` and regenerate any locally cached ABI
   wrappers. Update calls to `BondEscrow.reserveBountyPayout` to omit the recipient. Update
   `slashAuthorBond` and `refundAuthorBond` to omit the recipient as well; slashes always go to the
   immutable treasury and refunds always go to the recorded claim author. Replace direct
   outcome-status writes with `finalizeClaimResolution(claimId, replicationId)`, and replace
   `settleForecast(forecastId, resolutionStatus)` with
   `settleForecast(forecastId, latestResolutionDecisionId)`.
8. For every remote write-enabled service, configure `SP_RATE_LIMIT_BACKEND=postgres`. Keep
   privileged operator and cloud credentials out of any process marked `SP_PUBLIC_SERVICE=true`.
   Use `npm run indexer:rebuild:fresh` with `SP_REBUILD_DATABASE_URL` if cursor/hash verification
   detects a reorg; do not rewind the mixed operational read-model database in place.
9. Update signed `claim_create` payloads to use an immutable `ipfs://`, `ar://`, or `filecoin://`
   `artifactUri` and include `artifactSha256`, the SHA-256 digest of its exact bytes. The service
   resolves the canonical locator through a bounded gateway and verifies the digest before creating
   any onchain draft. Mutable HTTP(S) sources must enter through source ingestion and snapshotting.
10. Apply migration 028 before enabling public writes. It adds the expiring execution lease used to
    serialize retries for an accepted signed request without holding a database connection while
    artifact gateways or the chain are slow.

## Data semantics

- Existing agent budgets do not have an onchain `spentBudget` field. They remain part of the old
  deployment and are not copied automatically.
- Existing bounty reservations remain governed by the old escrow. New 0.3.0 reservations derive
  the recipient from the immutable replication record and cannot be released while unresolved.
- `BOUNTY_SETTLER_ROLE` can reserve and release deterministic replication payouts. Terminal
  cancellation and all author-bond movement remain under timelocked `ESCROW_ADMIN_ROLE`.
- Cancellation is terminal for a claim/replication reservation pair. It restores the claim's
  available bounty without transferring value.
- A claim remains `Draft` until its complete `requiredAuthorBond` is present in `BondEscrow`.
  Integrations must deposit directly from the author wallet and then submit a signed
  `claim_publish` request before requesting `Published`.
- A `ResolutionDecision` is an immutable copy of one resolved, claim-local replication. Every
  resolved replication may be recorded once, including evidence finalized after the claim has
  reached a stronger or terminal state. The decision stores its canonical status recommendation;
  the claim record advances only when that recommendation is a valid forward transition.
- `getLatestResolutionDecisionId` is the append-only evidence tail. The separate
  `getEffectiveResolutionDecisionId` pointer advances only when a decision establishes claim state;
  each forecast snapshots that pointer when committed and settlement requires a strictly newer
  effective decision. Later incompatible evidence cannot reverse settlement while leaving claim
  state unchanged, and a known decision cannot earn a hindsight bonus.
- Delegated service creation uses `createClaimOnBehalfWithRequestHash`; the signed request hash maps
  to exactly one claim id onchain, replacing bounded historical event scans as the duplicate barrier.
- `/write-config.authorBondWei` and `minimumAuthorBondWei` are equal aliases derived from the
  manifest floor. `authorBondWei` remains the client-default compatibility field.
- Migrations 023–028 add source-ingestion leases, indexer block checkpoints, reputation policy
  versioning, shared rate-limit buckets, canonical resolution-decision rows, and forecast decision
  links, plus public-write execution leases. Applied migration bytes are checksummed and may not be
  edited in place. Migration 027 also enforces at most one effective decision row per claim.

## Rollback

Rollback means restoring the previous service image digest, deployment manifest, and database
backup together. Never point a 0.2.x service at 0.3.0 addresses or a 0.3.0 service at the old
escrow. A rollback returns the operator to the old deployment; it does not undo transactions sent
to the new contracts.
